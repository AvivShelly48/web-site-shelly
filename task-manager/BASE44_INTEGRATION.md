# שילוב טאב "משימות" בתוך אפליקציית shelly-urban (Base44)

מסמך זה הוא ה־handoff להעברת פיצ'ר ניהול המשימות מהאפליקציה העצמאית
(`task-manager/`, Next.js) אל תוך אפליקציית **shelly-urban** הבנויה על **Base44**,
כטאב בתוך התוכנה הכללית שמשתמש במידע הקיים שם.

> **למה לא "להעתיק את הריפו":** האפליקציה הנוכחית היא Next.js עם DB מבוסס קובץ JSON.
> אפליקציית Base44 משתמשת בישויות (entities) מתוארחות, ב־SDK שלה ובאינטגרציות מובנות.
> לכן השילוב הוא **בנייה מחדש של הפיצ'ר בקונבנציות של Base44** — הלוגיקה וה־UI עוברים,
> שכבת הנתונים והפריימוורק לא.

---

## 1. מודל הנתונים

המשימות מתקשרות לישויות הקיימות באפליקציה (`Project`, `Contact`).
מוסיפים ישות חדשה אחת: `Task`.

### ישות חדשה: `Task`
| שדה | טיפוס | הערות |
|-----|-------|-------|
| `title` | string | כותרת המשימה (חובה) |
| `description` | string | תיאור חופשי |
| `project_id` | ref → Project | שיוך לפרויקט קיים (ראשי הפרקים) |
| `stage` | enum | שלב הפרויקט: `licensing` (רישוי), `construction` (בנייה), `form4` (טופס 4), `occupancy` (אכלוס) |
| `contact_id` | ref → Contact | איש הקשר האחראי / מקבל ההתראות |
| `frequency` | enum | `daily` / `weekly` / `monthly` |
| `due_date` | date | תאריך יעד |
| `status` | enum | `pending` (ממתינה), `in_progress` (בתהליך), `done` (הושלמה) |

> אם ל־`Project` אין כבר שדה שלב — `stage` יושב על ה־Task כדי לאפשר משימות בשלבים שונים
> באותו פרויקט. אם ל־`Project` כבר יש שלב נוכחי, אפשר לברירת־מחדל לרשת אותו.

### ישויות קיימות (לא לשנות, רק לקרוא/לקשר)
- `Project` — שם הפרויקט (`name`) משמש כראש הפרק בתצוגה.
- `Contact` — `name`, `email`, `phone` משמשים לשיוך משימה ולשליחת התראות.

---

## 2. מבנה המסך (טאב "משימות")

היררכיה: **פרויקט → שלב → משימות**

- **תצוגה ראשית:** רשימת משימות מקובצת לפי פרויקט, ובתוך כל פרויקט לפי שלב
  (רישוי / בנייה / טופס 4 / אכלוס).
- **פילטרים:** לפי פרויקט, לפי שלב, לפי תדירות (יומית/שבועית/חודשית), לפי איש קשר.
- **דשבורד קצר:** התקדמות כללית (% משימות שהושלמו) + התקדמות לפי פרויקט ולפי איש קשר.
- **עדכון inline:** שינוי סטטוס ושיוך איש קשר ישירות מהשורה/כרטיס.
- **מובייל:** כרטיסים (card list); דסקטופ: טבלה. (כפי שכבר מיושם ב־task-manager.)

---

## 3. התראות (מייל + הודעה)

לכל משימה אפשר לשלוח לאיש הקשר המשויך:

- **מייל:** דרך אינטגרציית הליבה של Base44 —
  `base44.integrations.Core.SendEmail({ to, subject, body })`.
- **הודעה (WhatsApp):** קישור `https://wa.me/<phone>?text=<encoded>` שנפתח מתוך האפליקציה,
  או אינטגרציית SMS/WhatsApp אם מוגדרת בחשבון Base44.

### תזכורות מרוכזות
מסך/פעולה לשליחת תזכורות לפי scope:
- `today` — משימות שהיעד שלהן היום
- `week` — משימות לשבוע הקרוב
- `overdue` — משימות שעבר זמנן וטרם הושלמו

ההודעה לכל איש קשר מקבצת את כל המשימות הפתוחות שלו להודעה אחת (להורדת עומס הוואטסאפים).

---

## 4. לוגיקה לשימוש חוזר (כבר קיימת בריפו הזה)

קבצים ב־`task-manager/lib/` שאפשר להמיר כמעט 1:1 ל־Base44 (החלפת קריאות ה־DB
ב־`base44.entities.*` ושמות שדות ל־snake_case):

- `lib/report.ts` — `buildReport({ frequency, ... })` בונה טקסט דוח לוואטסאפ/מייל.
- `lib/reminders.ts` — `buildEmployeeReminders(scope)` + `buildReminderMessage(...)`
  בונה תזכורת מקובצת לכל איש קשר. (להחליף "employee" ב־"contact".)
- `lib/types.ts` — `STATUS_LABELS`, `FREQUENCY_LABELS` + להוסיף `STAGE_LABELS`:
  `{ licensing: 'רישוי', construction: 'בנייה', form4: 'טופס 4', occupancy: 'אכלוס' }`.

---

## 5. צעדי ביצוע (כשתהיה גישה ל־shelly-urban)

1. לקרוא את מבנה הריפו: ישויות קיימות (`Project`, `Contact`), מבנה ה־pages/components,
   ואיך מוגדרת הניווט/הטאבים בלייאאוט.
2. להגדיר ישות `Task` בסכמת הישויות של Base44.
3. להוסיף עמוד `Tasks` ולחבר אותו לניווט הראשי כטאב.
4. לממש את התצוגה (פרויקט→שלב→משימות), הפילטרים, ועדכון inline.
5. לחבר התראות: SendEmail + קישור/אינטגרציית הודעה לאיש הקשר.
6. לממש תזכורות מרוכזות (today/week/overdue).
7. לבדוק מול נתונים אמיתיים מהאפליקציה ולדחוף לענף ייעודי ב־shelly-urban.
